# AgentGuard：面向 Code Agent 的安全执行沙箱

## 项目简介

AgentGuard 是一个面向 AI Code Agent 的安全执行沙箱，旨在解决 Code Agent 在自主读写文件、执行 Shell 命令和访问网络资源时可能带来的越权访问、误修改、误删除和数据外泄等风险。

在传统 Code Agent 使用场景中，Agent 往往可以直接操作本地工作区、运行命令或访问外部网络。一旦模型生成了错误操作、危险命令或被恶意 Prompt 诱导，就可能造成代码损坏、敏感文件泄露或宿主环境被破坏。

AgentGuard 通过容器隔离、最小权限控制、统一工具网关、动态提权审批、全链路审计和工作区回滚机制，为 Code Agent 提供一个受控、可观测、可恢复的执行环境。

项目的核心目标是：

> 让 Agent 可以完成真实代码任务，但所有高风险行为都必须被限制、审计和可回滚。

---

## 背景与问题

AI Code Agent 通常需要具备以下能力：

- 读取项目文件
- 修改代码
- 执行测试命令
- 安装依赖
- 访问外部文档或接口
- 根据执行结果继续修复代码

这些能力提升了 Agent 的自动化程度，但也引入了安全风险：

- Agent 可能读取工作区外的敏感文件
- Agent 可能执行高风险 Shell 命令
- Agent 可能误删或误改代码
- Agent 可能访问未授权的外部网络
- Agent 可能被 Prompt Injection 诱导泄露数据
- 用户难以追踪 Agent 到底执行了哪些操作
- 出现误操作后缺少快速恢复机制

AgentGuard 的目标不是阻止 Agent 执行任务，而是为 Agent 的执行过程增加安全边界。

## 核心设计思路

AgentGuard 将 Code Agent 的执行过程拆分为三层：

```text
┌────────────────────┐
│     Code Agent     │
│  决定下一步做什么    │
└─────────┬──────────┘
          │ 结构化工具调用
          ▼
┌────────────────────┐
│     AgentGuard     │
│ 鉴权 / 审批 / 审计  │
│ 快照 / Diff / 回滚  │
└─────────┬──────────┘
          │ 授权后的操作
          ▼
┌────────────────────┐
│  Container Sandbox │
│   隔离执行命令和代码 │
└────────────────────┘
```

其中：

- Code Agent 负责生成工具调用。
- AgentGuard 负责判断操作是否允许执行。
- Container Sandbox 负责在隔离环境中真正执行命令。

Agent 不直接访问宿主机文件系统、Docker、Shell 或网络资源，所有操作都必须经过 AgentGuard 的统一工具网关。

## 核心能力

### 1. 容器隔离执行

AgentGuard 为每个 Code Agent 任务创建独立的容器沙箱，将 Agent 的执行环境与宿主机隔离。

沙箱默认采用最小权限配置：

- 非 root 用户运行
- 禁止 privileged 模式
- 移除不必要的 Linux Capabilities
- 禁止 privilege escalation
- 限制 CPU、内存和进程数量
- 默认关闭网络访问
- 只挂载当前任务工作区
- 不挂载 Docker Socket
- 不挂载宿主根目录
- 不挂载宿主密钥和敏感配置文件

容器隔离负责提供底层安全边界，即使 Agent 生成了危险命令，也尽可能限制其影响范围。

---

### 2. 统一工具网关

Agent 不直接访问文件系统、Shell 或网络，而是通过 AgentGuard 提供的统一工具网关提交结构化工具调用。

示例：

```json
{
  "tool": "shell.exec",
  "arguments": {
    "argv": ["pytest", "tests"],
    "cwd": "/workspace",
    "timeout_seconds": 120
  }
}
```

统一工具网关负责：

- 接收 Agent 的工具调用
- 校验参数格式
- 规范化文件路径、命令参数和 URL
- 将请求转换为统一的 Action
- 调用策略引擎进行权限判断
- 根据结果决定拒绝、审批或执行
- 将执行结果返回给 Agent

计划支持的核心工具包括：

- `fs.read`：读取文件
- `fs.write`：写入文件
- `fs.delete`：删除文件
- `shell.exec`：执行 Shell 命令
- `net.request`：发起受控网络请求

---

### 3. 最小权限策略控制

AgentGuard 内置最小权限模型，默认拒绝未明确授权的操作。

策略引擎会根据以下信息进行判断：

- 操作类型
- 文件路径
- Shell 命令及参数
- 网络域名和端口
- 当前会话上下文
- 当前 Agent 身份
- 是否已有临时授权

策略引擎返回三种结果：

| 决策 | 含义 |
|---|---|
| `ALLOW` | 允许直接执行 |
| `DENY` | 拒绝执行 |
| `REQUIRE_APPROVAL` | 需要用户审批后才能执行 |

示例：

| 操作 | 决策 |
|---|---|
| 读取 `/workspace/src/main.py` | 允许 |
| 读取 `/etc/passwd` | 拒绝 |
| 执行 `pytest` | 允许 |
| 执行 `sudo` | 拒绝 |
| 执行 `rm -rf tests` | 需要审批 |
| 访问外部域名 | 需要审批 |

---

### 4. 动态提权审批

当 Agent 需要执行敏感操作时，例如删除文件、修改关键配置、执行高风险命令或连接外部网络，AgentGuard 不会立即执行，而是创建提权审批请求。

用户可以在审批界面中查看：

- 操作类型
- 完整命令或目标路径
- 触发的策略规则
- 风险等级
- 可能影响
- 授权范围

审批通过后，系统只会执行用户批准的那一次具体操作。

审批请求会绑定规范化后的 Action 指纹，避免出现以下问题：

- 用户批准的是 `rm tests/old_test.py`，Agent 实际执行 `rm -rf /`
- Agent 在审批通过后修改命令参数
- Agent 重复使用已经消费过的审批凭证
- Agent 使用过期审批继续执行操作

---

### 5. 全链路执行审计

AgentGuard 会记录 Agent 执行过程中的关键事件，包括：

- 会话创建
- 工具调用请求
- 权限策略判断
- 提权审批申请
- 用户审批结果
- Shell 命令执行
- 文件读写与删除
- 网络请求
- 工作区变更
- 回滚操作

审计事件示例：

```text
action_requested
policy_evaluated
approval_requested
approval_approved
execution_started
execution_completed
workspace_changed
rollback_completed
```

每条审计记录会包含：

- 时间
- 会话 ID
- Action ID
- 操作类型
- 风险等级
- 命中的策略规则
- 执行结果
- 相关文件变更

系统会对高风险操作进行标记，例如：

- 破坏性命令
- 敏感路径访问
- 外部网络连接
- 批量文件删除
- 工作区外路径访问
- 权限变更

敏感信息，例如 Token、Cookie、Authorization Header、密码和私钥，不会被原样写入审计日志。

---

### 6. Diff 展示与工作区回滚

AgentGuard 会在可能修改工作区的操作前创建快照，并在操作后计算文件变更。

系统支持展示：

- 新增文件
- 修改文件
- 删除文件
- 行级代码 Diff
- 每个操作对应的变更记录

当 Agent 发生误修改或误删除时，用户可以将工作区恢复到指定快照，降低 Agent 自主执行过程中的风险。

典型场景：

```text
Agent 执行 rm -rf tests
        ↓
系统要求用户审批
        ↓
用户批准后执行
        ↓
系统记录删除 Diff
        ↓
用户发现误删
        ↓
一键回滚到操作前状态
```

## 核心模块

### Tool Gateway

统一工具调用入口。Agent 的所有文件、命令和网络操作都必须先进入 Tool Gateway。

主要职责：

- 接收结构化工具调用
- 参数校验
- 路径、命令、URL 规范化
- 生成统一 Action
- 调用策略引擎
- 分发到审批、执行或拒绝流程

---

### Policy Engine

最小权限策略引擎，负责判断某个 Action 是否可以执行。

主要职责：

- 读取权限策略
- 判断文件路径访问范围
- 判断 Shell 命令风险
- 判断网络访问权限
- 返回 `ALLOW`、`DENY` 或 `REQUIRE_APPROVAL`
- 给出风险等级和命中规则

---

### Approval Service

动态提权审批服务，负责处理高风险操作。

主要职责：

- 创建审批请求
- 保存待审批 Action
- 计算 Action 指纹
- 接收用户批准或拒绝
- 生成一次性临时授权
- 防止审批篡改、重放和过期使用

---

### Sandbox Executor

容器沙箱执行器，负责在隔离环境中执行命令和代码。

主要职责：

- 创建和销毁容器
- 挂载任务工作区
- 执行结构化命令
- 获取 stdout、stderr 和 exit code
- 限制 CPU、内存、进程和网络
- 防止 Agent 访问宿主敏感资源

---

### Workspace Manager

工作区管理模块，负责快照、Diff 和回滚。

主要职责：

- 为每个任务创建独立工作区
- 在修改操作前创建 checkpoint
- 计算操作前后的文件变更
- 展示代码 Diff
- 恢复误修改或误删除的文件

---

### Audit Service

审计模块，负责记录 Agent 执行全过程。

主要职责：

- 记录工具调用
- 记录策略判断
- 记录审批结果
- 记录命令执行结果
- 记录文件和网络操作
- 标记高风险行为
- 支持按会话或 Action 追溯执行历史

---

### Network Controller

网络访问控制模块，负责限制 Agent 访问外部资源。

MVP 阶段采用默认无网络沙箱，所有网络请求必须通过 `net.request` 工具提交给 AgentGuard。

主要职责：

- 阻断容器直接联网
- 控制可访问域名、端口和协议
- 阻止 localhost、内网地址和云 metadata 地址
- 限制请求和响应大小
- 对外部访问进行审计

## 典型使用流程

1. 用户向 Code Agent 提交代码任务。
2. Agent 生成文件读取、代码修改或命令执行请求。
3. 所有工具调用进入 AgentGuard Tool Gateway。
4. Policy Engine 根据最小权限策略进行判断。
5. 普通操作直接在沙箱中执行。
6. 高风险操作进入用户审批流程。
7. 用户批准后，系统执行已审批的具体操作。
8. 执行前创建工作区快照。
9. 操作完成后生成 Diff 并写入审计日志。
10. 用户可查看操作时间线、高风险行为和文件变更。
11. 如发生误修改或误删除，可将工作区回滚到指定状态。

## 安全边界说明

AgentGuard 旨在降低 Code Agent 自主执行带来的风险，但不声称完全消除所有安全问题。

MVP 阶段的安全边界包括：

- Agent 无法直接访问宿主文件系统
- Agent 无法直接访问 Docker Socket
- Agent 无法直接联网
- Agent 的工具调用必须经过策略判断
- 高风险操作必须经过用户审批
- 工作区变更可以被审计和回滚

已知限制：

- Docker 容器不是强隔离虚拟机
- 工具级策略不能完全替代 OS 级隔离
- Shell 命令的语义分析无法做到完全准确
- 网络控制第一版仅支持通过 `net.request` 代理访问
- 操作级精细回滚可能存在冲突，需要后续增强

## 项目定位

AgentGuard 的目标不是重新实现一个完整的 Code Agent，而是为 Code Agent 提供一套安全执行基础设施。

项目重点包括：

- 限制 Agent 的文件、命令和网络访问范围
- 将所有工具调用统一纳入权限控制
- 对高风险操作引入用户审批
- 记录 Agent 执行过程中的完整行为
- 在误操作后支持快速恢复工作区

一句话概括：

> AgentGuard 是一个让 Code Agent 在受控、可审计、可回滚环境中执行任务的安全沙箱系统。
